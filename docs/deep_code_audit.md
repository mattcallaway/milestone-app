# Deep Code Audit Report (v2.6.0)

This document details findings from a comprehensive source code audit of the Milestone media resilience system.

## 1. Summary of Findings

| Severity | Count | Summary of Key Issues |
| :--- | :--- | :--- |
| **Critical** | 1 | Destination path flattening in Copier (risks data collision). |
| **High** | 2 | Queue missing `delete` implementation; Queue race condition in task start. |
| **Medium** | 3 | N+1 performance in sidecar reports; `signature_stub` dead field; Path normalization inconsistency. |
| **Low** | 2 | Missing Operation Enums; redundant color logic in frontend. |

---

## 2. Critical Issues

### [C-01] Destination Path Flattening
- **Location**: `services/api/src/copier.py:272`
- **Description**: The `create_copy_operation` builds the destination path using `os.path.basename(source_full)`. This strips the relative directory structure.
- **Risk**: If copying `C:\Movies\Action\Film.mp4` and `C:\Movies\Comedy\Film.mp4` to `D:\`, they will both attempt to write to `D:\Film.mp4`, leading to collision or unintended overwrites.
- **Fix**: Preserve relative paths from the root during copy planning.

---

## 3. High Issues

### [H-01] Missing `delete` Implementation in Queue
- **Location**: `services/api/src/queue.py:116`
- **Description**: The queue worker only handles `type == 'copy'`. Any `delete` operation (generated by Reduction or Retirement plans) is marked as `failed` with "Unknown operation type".
- **Risk**: Bulk reduction and drive retirement workflows are currently broken and cannot be completed.
- **Fix**: Implement the `delete` case in `process_operation`.

### [H-02] Queue Worker Race Condition
- **Location**: `services/api/src/queue.py:141-150`
- **Description**: `queue_worker` fetches pending ops and then creates tasks. There is a window between fetching and the task adding the ID to `active_ops`. If the loop runs twice (e.g. context switch during DB fetch), the same op could be fetched and started twice.
- **Risk**: Duplicate file operations if concurrency > 1 and the system is under heavy load.
- **Fix**: Add ops to `active_ops` *immediately* after fetching from the DB before the next loop iteration.

---

## 4. Medium Issues

### [M-01] N+1 Performance in Sidecar Reporting
- **Location**: `services/api/src/routers/sidecars.py:242`
- **Description**: The library-wide report iterates over every item and performs separate SQL queries for primary files and directory listings.
- **Risk**: Significant UI lag and backend timeout on libraries with >1000 items.
- **Fix**: Use batch JOIN queries to collect all sidecar data for the reported range in 1-2 calls.

### [M-02] Path Normalization Inconsistency
- **Location**: `services/api/src/scanner.py` and `services/api/src/copier.py`
- **Description**: `os.path.join` is used with raw strings. There is no consistent use of `os.path.normpath` or POSIX-normalisation before DB writes.
- **Risk**: Duplicate entries for the same file if accessed via different path separators (e.g. `C:\Data` vs `C:/Data`).
- **Fix**: Uniformly apply POSIX-style normalization (forward slashes) before DB insertion/query.

### [M-03] Dead Schema: `signature_stub`
- **Location**: `schema.sql:45`, `models.py:99`
- **Description**: Field is present in DB and API models but never assigned or updated by the scanner.
- **Fix**: Remove from schema or implement intended use-case (partial header sig).

---

## 5. Low Issues

### [L-01] Redundant UI Color Logic
- **Location**: `apps/desktop/src/screens/RiskScreen.tsx:22`
- **Description**: Color thresholds for RiskBar are hardcoded and duplicate the `RISK_COLORS` record.
- **Fix**: Reference `RISK_COLORS` from a central config.

### [L-02] Missing Operation Enums
- **Location**: `services/api/src/models.py`
- **Description**: Operations use raw string literals (`copy`, `delete`, `pending`) instead of Enums like `ScanState`.
- **Fix**: Centralize all status and type strings into Enums.
